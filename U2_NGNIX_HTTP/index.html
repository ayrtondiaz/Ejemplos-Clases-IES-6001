Slide 3

Métodos HTTP — Las Acciones de la Web

Cada método indica QUÉ querés hacer con el recurso

GET LEER

Pide un recurso al servidor. No modifica nada.

GET /productos → Devuelve la lista de productos
GET /usuarios/42 → Devuelve el usuario con ID 42

📖 Cargar páginas, imágenes, datos

POST CREAR

Envía datos para crear un recurso nuevo.

POST /productos → Crea un producto nuevo
POST /login → Envía usuario y contraseña

📝 Formularios, registros, subir archivos

PUT REEMPLAZAR

Envía datos para reemplazar completamente un recurso.

PUT /usuarios/42 → Reemplaza TODOS los datos del usuario 42

🔄 Actualizar un recurso completo

PATCH MODIFICAR

Modifica parcialmente un recurso (solo algunos campos).

PATCH /usuarios/42 → Cambia SOLO el email del usuario 42

✏️ Editar un campo específico

DELETE BORRAR

Elimina un recurso del servidor.

DELETE /usuarios/42 → Elimina el usuario 42

🗑️ Borrar registros, archivos, cuentas

🎯 Tip GET y POST son los más comunes. PUT, PATCH y DELETE los usaremos cuando hagamos APIs con FastAPI en la Unidad 4.
Slide 4

Códigos de Estado HTTP

¿Qué pasó con mi request? El servidor responde con un número que lo explica.

2xx — Éxito ✅

Todo salió bien

200
OK

Request exitoso. El servidor devolvió lo pedido.

201
Created

Se creó un recurso nuevo (respuesta a POST).

204
No Content

Éxito, pero sin contenido (respuesta a DELETE).

3xx — Redirección ↪️

El recurso se movió

301
Moved Permanently

La URL cambió para siempre. Usá la nueva.

304
Not Modified

El recurso no cambió. Usá tu caché.

4xx — Error del Cliente ❌

El cliente hizo algo mal

400
Bad Request

Datos inválidos o mal formateados.

401
Unauthorized

Necesitás autenticarte (login).

403
Forbidden

Estás autenticado pero no tenés permiso.

404
Not Found

El recurso no existe. URL incorrecta.

5xx — Error del Servidor 💥

El servidor falló

500
Internal Server Error

Algo se rompió en el servidor. Bug en el código.

502
Bad Gateway

El proxy (Nginx) no pudo contactar al backend.

503
Service Unavailable

Servidor sobrecargado o en mantenimiento.

🎯 Memorizar 200 (OK), 201 (Created), 400 (Bad Request), 401 (Unauthorized), 404 (Not Found), 500 (Server Error).

Redes LAN — Conceptos Fundamentales

IP · MAC · Puertos · DNS · localhost · TCP/IP

🏠

Dirección IP

Identificador numérico de cada dispositivo en la red.

192.168.1.100 — tu PC en la red local
127.0.0.1 — localhost (tu propia máquina)
🏷️

Dirección MAC

Identificador físico y único de la placa de red. No cambia.

A4:5E:60:B2:F1:3C
🚪

Puerto

Número que identifica un servicio específico en una máquina.

:80 — HTTP
:443 — HTTPS
:3000 — Live Server
:8000 — FastAPI / Backend
📖

DNS

Traduce nombres de dominio a direcciones IP.

google.com → 142.250.185.14
github.com → 140.82.121.4
🔄

TCP/IP

Protocolo que asegura que los datos lleguen completos y en orden.

TCP divide los datos en paquetes, los numera y los reensambla al llegar.
🏡

localhost

Tu propia computadora actuando como servidor.

http://localhost:3000
= http://127.0.0.1:3000
El navegador habla con tu propia máquina
🎯 Para la U4 Tu backend correrá en localhost:8000 y tu frontend en localhost:5173. Dos puertos, misma máquina.

Apache — Servidor Web + Virtual Hosts

Instalación · DocumentRoot · Virtual Hosts · Dominio local personalizado

📡

1. Recibe Request

Escucha en puerto 80 (HTTP) o 443 (HTTPS). Cuando un navegador pide una URL, el servidor recibe el request.

→
⚙️

2. Procesa

Busca el archivo en el disco (index.html, style.css). O ejecuta código backend (PHP, Python, Node).

→
📤

3. Envía Response

Devuelve el archivo con un código de estado (200, 404). El navegador renderiza el HTML/CSS/JS.

Paso 1: Instalar Apache en Ubuntu

Terminal 
# Instalar Apache
sudo apt update
sudo apt install apache2 -y

# Verificar que está corriendo
sudo systemctl status apache2
# ● apache2.service - The Apache HTTP Server
#   Active: active (running)  ← debe decir esto

# Abrir en el navegador:
# http://localhost
# Vas a ver: "It works!"

# ¿Dónde están los archivos del sitio?
# /var/www/html/     ← DocumentRoot por defecto

# Copiar tu sitio ahí:
sudo cp -r ~/mi-primer-sitio/* /var/www/html/
# Ahora http://localhost muestra TU sitio

Paso 2: Crear Virtual Host (tu propio dominio local)

Terminal 
# 1. Crear carpeta para el proyecto
sudo mkdir -p /var/www/mi-portfolio
sudo cp -r ~/mi-primer-sitio/* /var/www/mi-portfolio/
sudo chown -R $USER:$USER /var/www/mi-portfolio

# 2. Crear archivo de configuración
sudo nano /etc/apache2/sites-available/mi-portfolio.conf

Paso 3: Contenido del Virtual Host

Apache — /etc/apache2/sites-available/mi-portfolio.conf 
<VirtualHost *:80>
    ServerName mi-portfolio.local
    DocumentRoot /var/www/mi-portfolio

    <Directory /var/www/mi-portfolio>
        AllowOverride All
        Require all granted
    </Directory>

    ErrorLog  ${APACHE_LOG_DIR}/portfolio-error.log
    CustomLog ${APACHE_LOG_DIR}/portfolio-access.log combined
</VirtualHost>
ServerName → El dominio local que vas a escribir en el navegador
DocumentRoot → La carpeta donde están tus archivos HTML/CSS/JS
ErrorLog → Archivo donde se guardan los errores del sitio

Paso 4: Activar y probar

Terminal 
# 3. Agregar el dominio al archivo hosts
echo "127.0.0.1 mi-portfolio.local" | sudo tee -a /etc/hosts

# 4. Activar el sitio y reiniciar Apache
sudo a2ensite mi-portfolio.conf
sudo systemctl restart apache2

# 5. Abrir en el navegador:
# http://mi-portfolio.local
# ¡Tu portfolio está corriendo en Apache!

# ═══ Comandos útiles ═══
sudo a2ensite nombre.conf              # activar sitio
sudo a2dissite nombre.conf             # desactivar sitio
sudo systemctl restart apache2         # reiniciar
sudo systemctl stop apache2            # detener
sudo tail -f /var/log/apache2/error.log  # ver errores en vivo
🎯 Debug Si algo falla, revisá los logs: sudo tail -f /var/log/apache2/error.log te muestra los errores en tiempo real.

Apache vs Nginx — Comparativa

Dos servidores web, dos filosofías. En producción se usan juntos.

Apache

El más antiguo y extendido
  • Arquitectura: Un proceso/thread por conexión
  • Configuración: Por directorio (.htaccess)
  • Flexibilidad: Muy configurable, muchos módulos
  • Ideal para: Hosting compartido, desarrollo local
  • En la cursada: Lo usamos para Virtual Hosts ✓

Nginx

El más rápido para estáticos
  • Arquitectura: Basado en eventos (asíncrono)
  • Configuración: Centralizada (sin .htaccess)
  • Rendimiento: Superior para archivos estáticos y alto tráfico
  • Ideal para: Proxy reverso, producción de alto tráfico
  • En producción: Referencia de la industria

Tabla Comparativa Detallada

Criterio Apache Nginx
Arquitectura Procesos/Threads (fork por conexión) Eventos asíncronos (event loop)
Archivos estáticos Bueno Excelente (2-3x más rápido)
Configuración .htaccess por directorio Centralizada (nginx.conf)
Consumo de RAM Mayor (un thread por conexión) Menor (miles de conexiones en un proceso)
Uso en producción ~30% de sitios web ~34% de sitios web (líder)
Caso de uso ideal Hosting compartido, PHP, dev local Proxy reverso, alto tráfico, microservicios

Procesa lógica

Nginx recibe TODOS los requests. Los estáticos (CSS, JS, imágenes) los sirve directamente. Los dinámicos (APIs) los reenvía al backend.

🎯 En la industria Nginx suele estar delante como proxy reverso y Apache/Node/FastAPI detrás procesando la lógica de negocio.

Práctica — Chrome DevTools → Network

Tu herramienta #1 para debuggear problemas de red, APIs y carga de recursos

Pasos para Analizar Requests HTTP

1

Abrí cualquier sitio en Chrome

Ej: github.com o tu portfolio local

2

F12 → pestaña Network → recargá (F5)

Vas a ver TODOS los requests del navegador

3

Observá los requests

index.html (GET, 200) · style.css (GET, 200) · script.js (GET, 200) · logo.png (GET, 200)

4

Hacé click en un request para ver detalles

Headers: método, URL, status code · Response: contenido devuelto · Timing: cuánto tardó

5

Filtrá por tipo

Doc (HTML) · CSS · JS · Img · XHR (APIs/fetch)

6

Provocá un 404

Escribí una URL que no exista y observá el código 404 Not Found en la pestaña Network
